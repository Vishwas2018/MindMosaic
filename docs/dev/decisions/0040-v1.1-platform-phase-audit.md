# ADR-0040 — v1.1 Platform Phase Audit Findings

- Status: accepted
- Date: 2026-05-18
- Stage: v1.1-S5 chore close (pre-S6)
- Tags: backend, frontend, data, security, dx

## Context

Following the v1.1 platform phase exit (S1–S5 complete, Go verdict, 2026-05-18), a static
pre-S6 audit was conducted across the full platform phase deliverables. The audit covered
8 passes: P1 (Schema + RLS), P2 (API boundary validation + idempotency), P3 (error handling),
P4 (UI 5-state matrix + component contracts), P5 (accessibility), P6 (security posture),
P7 (performance patterns), P8 (production readiness). No code was modified — findings are
filed as issues and tracked for resolution.

The audit scope was bounded to v1.1-S1–S5 deliverables (migrations 0021–0022, content-svc
write paths, assessment-svc is_simulation chain, assignments-svc forwarding, StudentComposerForm
+ SimulationBanner + student route pages, teacher exam authoring pages). Pre-existing issues
from v1.0 stages (ISSUE-0023, ISSUE-0035, ISSUE-0036, ISSUE-0038) were noted but not re-filed.

## Findings summary

**0 Critical / 1 High / 5 Medium / 4 Low = 10 total new issues (ISSUE-0039–0048)**

### High (1)

- **ISSUE-0042** — Zod parse gap at content-svc and assessment-svc API boundaries
  (`content-svc/handlers.ts:840–848`, `assessment-svc/index.ts` createSession).
  Assertion-based input checks instead of `ItemCreateDTOSchema.parse()` / `CreateSessionRequestSchema.parse()`.
  **Pre-S6 blocker: YES.** S6 extends the content-svc write path; batch import must build on a
  Zod-validated write path.

### Medium (5)

- **ISSUE-0039** — Submit error does not discriminate 402 Upgrade Required
  (`StudentComposerForm.tsx:359–362`, `teacher/content/new/page.tsx:347–354`).
  Pre-S6 blocker: no.
- **ISSUE-0040** — SDK hooks missing staleTime causes refetch storms
  (`packages/sdk/src/hooks/content.ts` usePathways, `session.ts` useSessionState,
  `assignments.ts` useAssignmentsForClass). Pre-S6 blocker: no.
- **ISSUE-0041** — N+1 query patterns in assignments-svc handlers.ts (4 await-in-loop sites:
  publishAssignment, getAssignmentsForStudent, getAssignmentsForClass, getAssignmentTracking).
  Pre-S6 blocker: no.
- **ISSUE-0043** — assessment-svc /respond and /submit missing Idempotency-Key enforcement.
  Extends ISSUE-0023 pattern. Pre-S6 blocker: no.
- **ISSUE-0045** — Focus management missing on /practice and /exam-sim route entry.
  Pre-S6 blocker: no.

### Low (4)

- **ISSUE-0044** — assignments-svc flat error format inconsistent with other services.
- **ISSUE-0046** — `role="alert"` misuse on non-urgent form validation in StudentComposerForm.
- **ISSUE-0047** — Inline LoadingState in S4 teacher content pages instead of `@mm/ui` import.
- **ISSUE-0048** — PROJECT_STATE.md per-package test count discrepancy (docs-only; total 828 correct).

## Options considered

### Option A — Block S6 on all High + Medium findings

Fix ISSUE-0039–0043, 0045 before any S6 work. Ensures platform is clean before adding the import
pipeline layer. Pros: clean foundation. Cons: unnecessary delay — most Medium findings are
independent of S6's scope (SDK staleTime, N+1 in assignments-svc, focus management).

### Option B — Block S6 on pre-S6 blocker only (ISSUE-0042)

Fix only ISSUE-0042 (Zod gap in content-svc write path) before S6 impl. All other findings
carry as tracked issues, addressed at next natural touch point or before launch. Pros: targeted;
respects that S6 scope is the import pipeline, not a refactor of assessment-svc or SDK.
Cons: leaves 9 findings open at S6 start.

## Decision

**Option B.** Block S6 impl on ISSUE-0042 only. All other findings carry as open issues.

ISSUE-0042 is the only finding where S6 scope directly inherits the defect: S6 adds a batch
import endpoint that calls the same content-svc write path. Building on an assertion-validated
write path and then wrapping it in a Zod-validated batch layer would create a false security
boundary. ISSUE-0042 must be fixed at S6 prep and confirmed at S6 impl pre-push verification.

All other findings are independent of S6's deliverables (import pipeline, authoring spec templates,
ADR for pipeline patterns). They are tracked for launch-window remediation.

## Rationale

The audit surfaced no Critical findings and one High finding that is a genuine S6 blocker.
The pre-S6 Go verdict from the phase exit review (2026-05-18) stands — the platform is
structurally sound. The findings represent incremental quality debt, not architectural failures.

The Zod gap (ISSUE-0042) is a CLAUDE.md non-negotiable violation ("Zod validation at every API
boundary") on the write path that S6 directly extends. Fixing it at S6 prep is the lowest-cost
point: it requires touching `content-svc/handlers.ts` anyway (to add the batch endpoint), so the
fix is on the critical path rather than a detour.

P1 (schema + RLS): migrations 0021–0022 are structurally correct. Pattern G enforcement
(platform_admin + service-role) confirmed. No RLS gaps.

P6 (security): `Access-Control-Allow-Origin: *` on Edge Functions is pre-existing and acceptable
for JWT-protected endpoints (auth is in the bearer token, not the origin). TypeScript strict mode
confirmed across all 17 packages. No secrets in code.

P8 (production readiness): typecheck --force 17/17 green, lint 7/7 green, test 828/829 green.
Production build fails on `next/font` Google Fonts network timeout — pre-existing in this
development environment, not introduced in v1.1.

## Consequences

- Positive: 10 findings documented with severity, file:line, and pre-S6 blocker classification.
  No surprises at launch — known debt is tracked debt.
- Positive: S6 morning ritual has a concrete first task: fix ISSUE-0042 in content-svc at prep.
- Negative: 9 findings carry into S6 as open issues; launch-window remediation must close
  Medium findings before go-live.
- Follow-ups:
  - ISSUE-0042: ISSUE-0042 remediation lands as a dedicated pre-impl chore at S6 open — not
    folded into S6 content authoring commits. Atomic traceability for the boundary-validation fix.
    Confirm at S6 impl pre-push verification.
  - ISSUE-0039, 0043, 0045: fix before launch (submit error 402, idempotency, focus management).
  - ISSUE-0040, 0041: measure against k6 launch-window run; fix if p95 budgets exceeded.
  - ISSUE-0044, 0046, 0047, 0048: low — address opportunistically at next natural touch point.

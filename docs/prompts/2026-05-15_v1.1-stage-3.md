Stage 3 delivers the **administration layer** for exam-mode sessions: a backend
facility that locks a `mode='exam'` session into strict-test conditions — no
back-navigation, no per-item feedback until submit, deterministically scored.
Stage 2 added the **assembly layer** (`composer_params`). The two are orthogonal and
co-applicable: a composed simulation exam = session with both `composer_params` and
`simulation_params`.

**§N trap (caught at morning ritual, resolved Q-1.1-3.1):** Phase plan
`docs/dev/v1.1-phase-plan.md §S3` calls this "Simulation Exam Mode" and informally
claims "new session mode". But spec §18 Session Modes table (lines 2619–2628) defines
`mode='exam'` with Use Case verbatim **"Full practice exam simulation"** — the spec
already uses "simulation" as the description of mode='exam'. Spec row 'Challenge' is
"Timed competition, gamification" (leaderboard), a different product feature.
Therefore: spec-correct home = `mode='exam'`, NOT `'challenge'`, NOT a new enum. Same
§N trap structure as S2's "practice" vs "exam" trap, caught the same way.

## Constraints

1. **withIdempotency on POST /billing/checkout ONLY** — arch §1.8 explicit ("Required on: … /billing/checkout …"). ADR-0034 §Decision 3 distinction: checkout = REST idempotency with client-generated UUID in `Idempotency-Key` header; portal and cancel do NOT use the middleware. Q-43.1 self-resolve confirmed.

2. **Stripe-hosted Checkout redirect flow per Q-42.5 / ADR-0034 §Decision 5 (SAQ A)** — call `stripe.checkout.sessions.create()` with `mode: 'subscription'`; return `{ checkout_url, session_id }`. Never embed card data. No Stripe Elements.

3. **GET /billing/plans is PUBLIC — no auth** — arch §4.9 verbatim: "Public". SCREEN_SPECS §17: "public catalog". Route placed before Bearer-JWT block AND before service-role gate in `index.ts`. No `verifyBearer` call. Q-43.4 self-resolve confirmed.

4. **All other Stage 43 endpoints: Bearer-JWT via `verifyBearer`** — placed before service-role gate in `index.ts` (analytics-svc DEV-20260522-2 precedent confirmed). Service-role dual-path NOT applicable to user-facing billing endpoints.

5. **tenant_id from `auth.user.app_metadata?.['tenant_id'] as string`** — set by `handle_new_user()` trigger (migration 0001) for all roles. No `parent_student_link` join needed — billing is tenant-scoped, not student-scoped. Q-43.3 self-resolve confirmed.

6. **Idempotency-Key: client-generated UUID** (from `Idempotency-Key` request header) — per ADR-0034 contrast (NOT `stripe_event_id`). Header parsed as `req.headers.get('Idempotency-Key') ?? ''`.

7. **PLAN_CATALOG single source of truth** — one `const PLAN_CATALOG` object in `handlers.ts`, imported by both `handleGetPlans` and `handleCreateCheckout`. Stripe Price IDs from env vars: `STRIPE_PRICE_ID_STANDARD_MONTHLY`, `STRIPE_PRICE_ID_STANDARD_YEARLY`, `STRIPE_PRICE_ID_PREMIUM_MONTHLY`, `STRIPE_PRICE_ID_PREMIUM_YEARLY`. Read once at handler-call time (not module scope, to allow env override in tests). Q-43.6 self-resolve confirmed. Grep verification: catalog data structure appears exactly once in billing-svc tree (1 declaration, 2 consumer sites).

8. **Role enforcement on checkout / portal / cancel: parent or org_admin only** — return 403 `FORBIDDEN` for student role. Roles `subscription` and `invoices` allow any authenticated role (student, parent, teacher, org_admin, platform_admin).

9. **GET /billing/subscription: synthetic free-tier on missing row** — if `maybeSingle()` returns null, return `{ tier: 'free', is_active: true, started_at: new Date().toISOString(), current_period_end: null, cancel_at: null, canceled_at: null, stripe_subscription_id: null }`. No 404. Q-43.5 self-resolve confirmed.

10. **GET /billing/invoices: LIMIT 50 + `truncated: boolean`** — `ORDER BY invoiced_at DESC LIMIT 51`; if 51 rows returned, set `truncated: true` and return first 50. Inline comment: `// ISSUE-0033: v1 LIMIT 50 + truncated flag; cursor pagination deferred to v1.1.` Q-43.2 Option A confirmed. `InvoicesResponseSchema = { invoices: z.array(InvoiceDTOSchema), truncated: z.boolean() }`.

11. **POST /billing/subscription/cancel: `?undo=true` query param** — body may be empty. `undo = url.searchParams.get('undo') === 'true'`. Call Stripe to schedule `cancel_at_period_end` (undo=false) or remove schedule (undo=true). Return `CancelResponseSchema = { cancel_at: z.string().datetime().nullable(), is_active: z.boolean() }`.

12. **Stripe client passed to new handlers** — `index.ts` already initialises `const stripe = new Stripe(...)`. New handler functions receive `stripe` as a parameter (same pattern as `handleStripeWebhook`). Do NOT re-initialise Stripe in handlers.ts.

13. **Bearer-JWT routes placed before service-role gate** — analytics-svc pattern confirmed. GET /billing/plans before all gates; remaining 5 endpoints (checkout, portal, subscription, cancel, invoices) between webhook block and service-role gate.

14. **ADR-0029 SDK prefix** — all SDK hooks use `/billing-svc/billing/...` client path. Grep: `grep -n '/billing-svc/billing/' packages/sdk/src/hooks/billing.ts` → ≥6 hits.

15. **No `any` in production code** — TypeScript strict. Zod parse at every API boundary. `BillingDbClient` interface already defined in handlers.ts; extend structurally if new DB operations needed.

## Deliverables

### 1. `supabase/functions/billing-svc/handlers.ts` (amend)
Add 6 handler functions:
- `handleGetPlans(opts: { planCatalog: PlanCatalog }): HandlerResult<PlanCatalogDTO>` — returns PLAN_CATALOG. No DB read, no auth.
- `handleCreateCheckout(opts: { body: CheckoutRequest, idempotencyKey: string, tenantId: string, stripe: StripeClient, client: BillingDbClient, traceId: string }): Promise<IdempotencyResult<CheckoutResponseDTO>>` — calls `withIdempotency`; inside handler: look up or create `billing_customer`, call `stripe.checkout.sessions.create()`, return `{ checkout_url, session_id }`.
- `handleCreatePortalSession(opts: { tenantId: string, stripe: StripeClient, client: BillingDbClient, traceId: string }): Promise<HandlerOutcome<{ portal_url: string }>>` — look up `billing_customer`, call `stripe.billingPortal.sessions.create()`, return `{ portal_url }`.
- `handleGetSubscription(opts: { tenantId: string, client: BillingDbClient }): Promise<HandlerOutcome<SubscriptionDTO>>` — SELECT from `subscription` WHERE `tenant_id`; on null return synthetic free-tier SubscriptionDTO (Q-43.5).
- `handleCancelSubscription(opts: { tenantId: string, undo: boolean, stripe: StripeClient, client: BillingDbClient }): Promise<HandlerOutcome<CancelResponseDTO>>` — look up `subscription.stripe_subscription_id`; call Stripe to set/unset `cancel_at_period_end`; UPDATE local `subscription.cancel_at`; return updated `{ cancel_at, is_active }`.
- `handleGetInvoices(opts: { tenantId: string, client: BillingDbClient }): Promise<HandlerOutcome<InvoicesResponseDTO>>` — SELECT LIMIT 51 ORDER BY `invoiced_at DESC`; `truncated = rows.length === 51`; return first 50 (ISSUE-0033 marker).

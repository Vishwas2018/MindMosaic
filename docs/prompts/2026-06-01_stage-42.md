## Constraints

### Stripe SDK
- Import: `https://esm.sh/stripe@16` — matches existing esm.sh pattern
  (jobs-worker uses `https://esm.sh/@supabase/supabase-js@2`).
- Version must be pinned (no @latest).

### Webhook signature verification (≤300ms budget)
- Raw body: `const rawBody = await req.text()` BEFORE any JSON parse or DB write.
- Signature header: `stripe-signature`.
- Call: `stripe.webhooks.constructEvent(rawBody, sig, STRIPE_WEBHOOK_SECRET)`.
- On invalid signature: return HTTP 400 immediately + write billing_event row
  with `processing_error='invalid_signature'` (arch §1.3 verbatim: "reject
  within 300 ms if signature invalid").
- MUST: no `req.json()` call before signature verification — raw body stream
  consumed once.

### Event idempotency (Q-42.4 — explicit divergence from ISSUE-0023)
- Mechanism: `INSERT INTO billing_event (...) VALUES (...) ON CONFLICT (stripe_event_id) DO NOTHING`.
- Return HTTP 200 on duplicate (Stripe expects 2xx; non-2xx triggers retry).
- Do NOT use withIdempotency middleware (_shared/idempotency.ts) — that is for
  REST endpoints with client-generated Idempotency-Key headers.
- ADR-0034 documents this contrast explicitly.
- ISSUE-0032 inline comment at STRIPE_WEBHOOK_SECRET load site in
  billing-svc/handlers.ts:
  `// ISSUE-0032: single-secret; v1.1 needs dual-secret rotation window`

### Stripe key strategy (Q-42.1)
- Single `STRIPE_SECRET_KEY` env var; `sk_test_...` vs `sk_live_...` prefix
  determines mode at deploy time.
- Separate `STRIPE_WEBHOOK_SECRET` per environment (always different per Stripe).
- No `STRIPE_SECRET_KEY_TEST`, `STRIPE_SECRET_KEY_LIVE`, or `STRIPE_MODE`
  branching in code.
- grep -n 'STRIPE_SECRET_KEY' billing-svc → ≥1 hit.
- grep -n 'STRIPE_SECRET_KEY_TEST\|STRIPE_SECRET_KEY_LIVE' billing-svc → 0 hits.

### RLS pattern (Pattern G)
- All 4 billing tables: service-role only, no anon/authenticated direct access.
- Each table: `ALTER TABLE ... ENABLE ROW LEVEL SECURITY;` + policy that
  allows only service-role.
- grep -n 'ENABLE ROW LEVEL SECURITY' 0018_billing.sql → ≥4 hits.

### job_type naming (Q-42.6)
- Job name: `pipeline.feature_flag_propagate` (arch §11.2).
- Owning service: billing-svc.
- Route: `POST /billing/pipeline/flag-propagate`.
- Stage 42 stubs handler with HTTP 200 + admin_action_log entry +
  `// Stage 44 pending` marker. NOT a silent no-op.
- DEV_PLAN shorthand `pipeline.billing_event_apply` must NEVER appear as a
  job_type string in code or new docs.
- grep -n 'pipeline.feature_flag_propagate' jobs-worker/ → ≥1 hit.
- grep -n 'pipeline\.billing_event_apply' supabase/ docs/ → 0 hits in new files.

### Webhook handler flow (Q-42.6 resolution)
1. `const rawBody = await req.text()`
2. Verify `stripe-signature` → constructEvent → on failure: 400 + billing_event row
3. `INSERT INTO billing_event ... ON CONFLICT (stripe_event_id) DO NOTHING`
4. Check `count = 0` (duplicate) → return 200 immediately (no-op)
5. Sync: update `subscription` + `billing_customer` based on event type
6. Enqueue `pipeline.feature_flag_propagate` job in `job_queue`
7. Return 200

### Stage 44 stub constraint
- `POST /billing/pipeline/flag-propagate` handler body in Stage 42:
  200 response + `admin_action_log` entry + explicit `// Stage 44 pending` marker.
  Not a silent no-op — must be auditable that the job was received.
- grep -n '// Stage 44 pending' billing-svc/ → ≥1 hit.

### Migration 0018 deferred-validation pattern
- Written but not run against real Postgres (no Docker in sandbox).
- Follow same pattern as 0012–0017 (documented in deployment.md).
- deployment.md billing section must include:
  - STRIPE_* env vars
  - BILLING_SVC_URL (jobs-worker dispatch)
  - Elevated-criticality note: payment data — billing_event is immutable audit
    log; once a billing_event is written it must not be deleted even if the
    related subscription row is corrected. Financial records retain 7 years
    (arch §8.3).
  - Migration 0018 deploy order: migration → then billing-svc deploy. (Mirror
    0017 deploy-order constraint structure.)
  - Rollback strategy: DROP TABLE order respects FK dependency:
    billing_event → invoice → billing_customer → subscription. All DROP CASCADE
    will cascade to downstream; production rollback requires data preservation
    policy (backup before DROP; Stripe is source of truth for recovery).

### Stripe seed script
- `scripts/stripe-seed.sh` — Stripe CLI commands to create test-mode Products
  and Prices: Standard (monthly AUD, yearly AUD) + Premium (monthly AUD, yearly
  AUD). Four price IDs output for use in .env.local.
- Include AU GST note: Stripe Tax must be configured in Stripe Dashboard for AU
  GST 10% to activate automatically. Tax is not configured via CLI in v1.

### 50-event replay test — deterministic event IDs
- Replay test seeds 50 events with deterministic IDs:
  `evt_test_0001` … `evt_test_0050`.
- Do NOT use crypto.randomUUID() for test event IDs — replay verification
  requires determinism.
- Send 50 events, verify 50 billing_event rows.
- Replay same 50, verify still exactly 50 rows (ON CONFLICT no-op).
- Invalid signature test: verify 400 + billing_event row with
  processing_error='invalid_signature'.
- grep -n 'evt_test_0001\|evt_test_0050' billing-svc/__tests__/ → ≥2 hits.

### Test target floor
≥10 new tests (50-event replay counts as one test setup + assertion bundle,
not 50 separate tests).

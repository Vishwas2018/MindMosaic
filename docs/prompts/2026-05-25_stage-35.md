## Context

Stage 35 adds `POST /orchestration/overrides` and `DELETE /orchestration/overrides/{id}` to `orchestration-svc`. Both are synchronous parent/teacher Bearer-JWT actions — not async pipeline jobs (no new jobs-worker route; ADR-0031 unchanged).

The `plan_override` table exists in migration 0005 with RLS enabled (Pattern A — service-role writes bypass RLS). The consumption path in `processOrchestratorReplan` ([handlers.ts:364–582](../../supabase/functions/orchestration-svc/handlers.ts)) already loads and applies overrides. Stage 35 adds only the create/delete surface — the consumption path is untouched.

**T1 evidence pins (verbatim from pre-reads):**

- `override_type` enum (`PlanOverrideTypeSchema`, [shared.ts:116](../../packages/types/src/shared.ts)): `z.enum(['pin_skill', 'dismiss_recommendation', 'override_plan_item'])`
- `expires_at` default: `now() + 14 days` per Spec §16.6.1 line 2504; constant `OVERRIDE_DEFAULT_EXPIRY_DAYS = 14` at [handlers.ts:44](../../supabase/functions/orchestration-svc/handlers.ts)

- `actor_id` FK: `REFERENCES user_profile(id) ON DELETE RESTRICT` — must be populated from `caller.userId`
- Stage 35 types in scope: `pin_skill` + `dismiss_recommendation` only (`override_plan_item` deferred per DEV_PLAN)
- Spec §16.6.1 audit requirement verbatim: "Every override creation and expiry writes an entry to `intelligence_audit_log` with `layer='L9_override'` so plan changes remain fully traceable."
- Self-supersession verbatim (Spec §16.6.1 line 2510): "Overrides of the same `type` + `target` replace each other — creating a new `pin_skill` for the same `skill_id` extends the expiry and updates priority rather than stacking."

**Response DTO (Q-35.1 Modified Option A):**

## Constraints

1. **No touch on consumption path**: `processOrchestratorReplan` lines 364–582 in [handlers.ts](../../supabase/functions/orchestration-svc/handlers.ts) is read-only from Stage 35. Only `createOverride` + `deleteOverride` are new.

2. **No new jobs-worker route**: POST/DELETE /overrides are synchronous Bearer-JWT actions. ADR-0031 route table unchanged. No fifth amendment.

3. **No outbox event from override create/delete**: Spec §16.6.1 specifies `intelligence_audit_log` writes for override lifecycle only. The student notification path is: replan → `plan_updated` outbox event → notification. Override creation triggers the *next* replan naturally on the student's next session submission (R9 finding).

4. **Role gate (Q-35.2 Option A)**: `['parent', 'teacher', 'tutor', 'org_admin', 'platform_admin']`. Per role, link verification:
   - `parent` → `student_id ∈ parent_student_link` query (caller's linked children)
   - `teacher` / `tutor` → `student_id ∈ class_group + class_student` join (caller's class roster)
   - `org_admin` → `tenant_id` matches caller's `app_metadata.tenant_id`
   - `platform_admin` → no link check (via service-role dual-path or direct svc header)
   - Any other role → 403 FORBIDDEN

5. **DELETE authorization**: `actor_id = caller.userId` OR `caller.role IN ('org_admin', 'platform_admin')`. Mismatched actor + non-admin → 403.

6. **Self-supersession (Q-35.3 Option A)**: Application-layer SELECT + UPDATE/INSERT. Deterministic key:
   - `pin_skill`: `(student_id, type='pin_skill', target->>'skill_id')`
   - `dismiss_recommendation`: `(student_id, type='dismiss_recommendation', target->>'recommendation_key')`
   - If active row found (`.gte('expires_at', now())`): UPDATE `expires_at` + `priority` (pin_skill only). Return updated row.
   - If not found: INSERT new row.

7. **override_plan_item rejection (Q-35.4 Option A)**: Return `400` with `{ error: 'UNSUPPORTED_TYPE', message: 'override_plan_item deferred in v1; not yet supported.' }`.

8. **Audit log writes (Spec §16.6.1)**:
   - POST create: `event_type='override_created'`, `layer='L9_override'`, `algorithm_version=L9_ALGORITHM_VERSION`, `input_snapshot={ override_id, type, target, expires_at, actor_id }`, `output={ success: true }`
   - DELETE: `event_type='override_deleted'`, `layer='L9_override'`, `algorithm_version=L9_ALGORITHM_VERSION`, `input_snapshot={ override_id, deleted_by_actor_id }`, `output={ success: true }`
   - No dedup guard (user-initiated actions, not retried async jobs — divergence from L9 replan pattern per R11 finding).

9. **Route placement**: Both route blocks placed BEFORE the service-role gate at [index.ts:147](../../supabase/functions/orchestration-svc/index.ts) per Stage 31/33/34 pattern. Dual-path auth: service-role-as-platform_admin OR verifyBearer JWT.

10. **RLS**: Pattern A — service-role writes bypass RLS. Authorization enforced entirely at handler layer. No migration needed (no new tables, no RLS policy changes).

11. **T2-tightened**: Mid-impl self-resolve Qs filed in same work session as resolving code.

12. **T3 Option 3 hybrid**: Q-35.1 + Q-35.2 were round-trip (DTO shape + auth model); Q-35.3 + Q-35.4 were self-resolve (tight implementation details). All four filed in this prep commit before any handler code.

13. **Stale-comment guard**: Copy-pasting from Stage 31/34 handler scaffolds — retitle `event_type` strings, `algorithm_version` usage, log labels to override-handler context. No stale `session_id`, `plan_id`, or replan-specific payload keys in override handler bodies.

14. **`no any` rule**: Zero `any` type annotations in new production code. Use `unknown` + assertion pattern where needed (Stage 34 `as unknown as T` precedent).
